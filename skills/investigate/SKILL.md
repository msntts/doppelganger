# /investigate

原因究明に徹する。修正は行わない。調査で得た知見を PLAN.md の Hotfix セクションに残し、`/execute` に引き継ぐことがゴール。

## トリガー条件

以下のいずれかに該当するとき、このスキルを使う：

- コードや現在見えているログだけでは原因を特定できない
- ユーザーと Claude の認識に齟齬があり、コードを読んでも解消されない
- 「何が起きているか」を把握するために実際の実行ログが必要と判断した

**外部サービスの不具合**（MCP サーバー・ブラウザ自動化・外部 API）は、コードを変更する前に必ずこのスキルの手順を踏む。盲目的なリトライ・「とりあえず試す」式の修正はしない。

`/execute` から移行する場合、または単独で起動した場合どちらでも使える。

---

## 移行手順

1. ユーザーに移行することを宣言する
   ```
   🔍 ログの確認が必要と判断しました。debug ブランチで調査します。
   ```
2. 現在のブランチから `debug/{調査内容の一言}` ブランチを作成する
3. PLAN.md が存在する場合、Hotfix セクションに調査タスクを追記してコミットする

---

## 調査フロー

1. **仮説を立てる** — 何が原因である可能性があるかを列挙する
2. **ログを仕込む** — stdout / stderr / ファイルへの書き込みでログを追加する（ユーザーにコピペを求めない）
3. **実行して観察する** — ログを取得・分析する
4. **仮説を検証する** — 観察結果と仮説を照合し、絞り込む
5. **原因を特定する** — 特定できたら修正プランを立てる

ループが必要な場合（1回の実行で原因が絞りきれない場合）は手順 1〜4 を繰り返す。

---

## 収束手順

原因が特定できたら：

1. PLAN.md の Hotfix セクションに修正プランを追記してコミットする
2. デバッグ中に作成した git 管理外ファイルを削除する
3. デバッグ中にインストールしたパッケージを削除する
4. デバッグ用のログコードを削除する
5. debug ブランチを削除する
6. 元のブランチに戻る
7. ユーザーに報告する：
   ```
   🔍 調査完了 — {原因の一文}

   {原因の詳細}

   修正プランを PLAN.md の Hotfix に追記しました。/execute で続けてください。
   ```

---

## 注意事項

- **修正はしない** — 原因究明のみ。修正は `/execute` で行う
- デバッグ用コードはメインブランチに持ち込まない
- `debug/` ブランチでは `git push` しない
- 原因が特定できない場合はユーザーに現状を報告し、追加情報を求める
